# 需求澄清优先原则

在开始实现任何功能之前，必须先充分理解和澄清用户需求。

## 核心原则

**永远不要根据模糊的需求直接开始编码。** 当用户提出的需求不够具体时，必须先通过提问来明确细节。

## 何时需要澄清需求

当用户的需求包含以下特征时，必须先提问澄清：

1. **功能描述模糊**
   - 例如："做一个画板程序"、"创建一个表单"、"添加一个列表"
   - 缺少具体的功能点、交互方式、数据结构等细节

2. **技术选型不明确**
   - 没有指定使用的库或框架
   - 没有说明是否需要特定的技术实现

3. **UI/UX 要求不清晰**
   - 没有描述界面布局
   - 没有说明交互方式和用户体验要求

4. **数据和状态管理未定义**
   - 数据来源不明确
   - 状态管理方案未指定
   - 数据持久化需求不清楚

5. **边界条件和约束未说明**
   - 性能要求
   - 浏览器兼容性
   - 响应式设计需求
   - 可访问性要求

## 澄清需求的方法

### 1. 分类提问

将需求拆解为不同维度，逐一确认：

**功能维度**
- 核心功能有哪些？
- 每个功能的具体行为是什么？
- 功能之间如何交互？

**技术维度**
- 使用什么技术栈？
- 是否有性能要求？
- 是否需要考虑兼容性？

**UI/UX 维度**
- 界面布局是什么样的？
- 用户如何操作？
- 需要什么样的视觉反馈？

**数据维度**
- 数据从哪里来？
- 数据如何存储？
- 是否需要持久化？

### 2. 提供选项

当有多种实现方案时，列出选项让用户选择：

```
我理解你想创建一个画板程序。为了更好地实现，我需要确认几个关键点：

1. 绘图功能：
   - [ ] 自由绘制（画笔）
   - [ ] 几何图形（矩形、圆形、线条等）
   - [ ] 文字输入
   - [ ] 橡皮擦

2. 技术实现：
   - [ ] HTML5 Canvas
   - [ ] SVG
   - [ ] 第三方库（如 Fabric.js、Konva.js）

3. 功能特性：
   - [ ] 撤销/重做
   - [ ] 颜色选择
   - [ ] 画笔粗细调整
   - [ ] 导出图片
   - [ ] 保存/加载画板内容

请告诉我你需要哪些功能，我会据此创建合适的实现方案。
```

### 3. 参考案例

提供类似产品的参考，帮助用户明确需求：

```
你想要的画板程序类似于以下哪种？
- 简单涂鸦板（如 Windows 画图）
- 专业绘图工具（如 Excalidraw）
- 白板协作工具（如 Miro）
- 手写笔记应用（如 GoodNotes）
```

## 实施步骤

1. **接收需求** → 评估需求的明确程度
2. **识别模糊点** → 列出需要澄清的方面
3. **提出问题** → 使用分类提问、选项列表或参考案例
4. **等待确认** → 不要在用户回复前开始编码
5. **再次确认** → 用自己的话复述需求，确保理解一致
6. **开始实现** → 只有在需求完全明确后才开始编码

## 反面示例（禁止）

❌ 用户说："做一个画板程序"
→ 直接创建一个包含各种功能的复杂画板应用

❌ 用户说："添加一个表单"
→ 猜测表单字段和验证规则，直接实现

❌ 用户说："优化这个组件"
→ 在不了解性能瓶颈的情况下随意修改

## 正面示例（推荐）

✅ 用户说："做一个画板程序"
→ 提问澄清：绘图功能、技术选型、特性需求、数据持久化等

✅ 用户说："添加一个表单"
→ 提问澄清：表单用途、字段类型、验证规则、提交处理等

✅ 用户说："优化这个组件"
→ 提问澄清：具体的性能问题、优化目标、可接受的权衡等

## 例外情况

以下情况可以直接实现，无需过度提问：

1. **需求已经非常具体**
   - 用户提供了详细的功能列表
   - 包含了技术选型和实现细节
   - 有明确的验收标准

2. **修复明确的 Bug**
   - 错误信息清晰
   - 复现步骤明确
   - 预期行为已知

3. **简单的代码调整**
   - 修改变量名
   - 调整样式
   - 更新文案

4. **遵循已有模式**
   - 项目中已有类似实现
   - 只是复用现有模式

## 总结

**记住：花 5 分钟澄清需求，胜过花 1 小时实现错误的功能。**

用户的时间很宝贵，通过提前澄清需求，可以：
- 避免返工和浪费时间
- 提供更符合预期的解决方案
- 建立更好的沟通和信任
- 展现专业性和责任心
